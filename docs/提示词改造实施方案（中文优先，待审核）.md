# 提示词改造实施方案（中文优先，待审核）

## 1. 目标与约束

本方案用于落实你提供的《提示词修改方案》，并结合当前 Professor 代码现状做工程化落地。

硬约束：

- 本轮仅改中文路径提示词，英文分支暂不改。
- 不引入“核心功能兜底继续”的策略：关键校验失败必须报错/重试，不可静默放过。
- 先出方案文档审核，通过后再改代码。

## 2. 现状结论（与本次方案直接相关）

当前代码位置：`lib/openai/videoInterpretation.ts`

1. 解读两阶段：

- `generateCoverageMap`：先产出 `SUMMARY + POINTS`
- `generateFullArticle`：再根据 `POINTS` 写全文

2. 当前“章节 1 = Integrated Interpretation”的触发机制：

- `splitArticleIntoSections` 按 `## ` 标题切段。
- 若模型在第一个 `##` 前输出正文，这段会被收集并以默认标题 `Integrated Interpretation` 入章。
- 这在产品行为上可视为 bug（格式不符合预期），不是期望行为。

3. 当前 coverage 通过门槛过低：

- 只要 `POINTS >= 1` 就通过，这与“章节策划 4-8/3-5”的目标不一致。

## 3. 本次改造范围（仅中文）

### 3.1 覆盖阶段提示词（Coverage）

文件：`lib/openai/videoInterpretation.ts`  
函数：`generateCoverageMap(...).buildPrompt(...)` 中文分支

改造原则：

- 保留输出契约标签 `SUMMARY:` / `POINTS:`（兼容现有解析）。
- 将语义从“信息点列表”提升为“章节策划议题”。
- 点数要求按模式明确写死：
  - detailed：4-8
  - concise：3-5
- 强调“标题有辨识度 + 含关键论据”。

预期效果：

- POINTS 粒度更“章节化”，减少碎片化知识点。

### 3.2 文章阶段提示词（Article）

文件：`lib/openai/videoInterpretation.ts`  
函数：`generateFullArticle(...).buildPrompt(...)` 中文分支

改造原则：

- 加入硬格式约束：
  - 第一行必须是 `## ` 标题
  - 第一个 `##` 前禁止任何正文、导语、寒暄
- 章节数约束：
  - detailed：4-8
  - concise：3-6
- 保留“专栏感、可读性、逻辑过渡、引用/加粗”风格要求。

预期效果：

- 显著降低 `Integrated Interpretation` 作为首章的概率。

## 4. 工程化补强（确保不靠运气）

仅改提示词还不够稳定，需增加程序化校验。

### 4.1 Coverage 结果校验（关键）

文件：`lib/openai/videoInterpretation.ts`  
位置：`generateCoverageMap` 在 `parseCoverageText` 之后

新增规则：

- detailed 模式：`POINTS` 至少 4 条，最多 8 条（超过可截断或报错重试，待你确认）
- concise 模式：`POINTS` 至少 3 条，最多 5 条（同上）

失败处理：

- 触发一次“格式/数量纠偏重试提示”再调用模型。
- 重试后仍失败：抛错终止本次解读流程（不静默降级）。

### 4.2 Article 结构校验（关键）

文件：`lib/openai/videoInterpretation.ts`  
位置：`generateFullArticle` 返回文本后、`splitArticleIntoSections` 前

新增规则：

- 首个非空行必须匹配 `^##\s+`
- 按 `##` 解析出的章节数需满足模式范围：
  - detailed：4-8
  - concise：3-6

失败处理：

- 触发一次“仅修复格式，不改语义”的重试。
- 仍失败：抛错，流程失败并返回错误状态。

说明：

- 这一步是为了解决“有时模型不听话”的不稳定性。
- 符合你“核心步骤失败就报错，不要兜底继续”的要求。

## 5. 与现有解析兼容策略

保持不变：

- 不改数据库结构

会改变：

## 6. 仅中文改造的具体边界

具体表现：

测试样本：

- 以 `docs/tt.txt` 为主样本，模式分别跑 `detailed` / `concise`。

通过标准：
